# KMA 중기예보 통합 설계안 (구현 전 검토용)

작성: 2026-04-20 정민

## 배경

- **ECMWF S2S (16~46일)**: ECMWF 키 있으나 S2S 데이터셋 **접근 권한 없음(403)** → 포기
- **OpenWeather API**: 무료 5일, 유료 30일 — 현재 Open-Meteo(16일) 대비 투자 가치 낮음 → 비채택
- **평년값 피처(temp_anomaly)**: 2026-04-20 실험 결과 overall MAE -2.5%, winter MAE +6% → 학습 피처로는 롤백. `clim_*`은 CSV에 남겨 장기 예측 NaN fallback용으로만 유지
- 남은 현실적 선택지: **기상청 KMA 중기예보**로 +4~+10일 구간 정확도 개선

## 목표

**Open-Meteo ECMWF(0~15일, 글로벌 모델)** 단독 사용을 **Open-Meteo(0~3일, 11~15일) + KMA 중기예보(4~10일)** 하이브리드로 전환하여 한반도 특화 정확도를 확보.

## 현재 상태 (검증 완료)

### KMA 중기예보 API 실측

- 엔드포인트 2종 정상 동작 (HTTP 200, 2026-04-20 테스트):
  - `getMidTa` — +4~+10일 최저/최고 기온
  - `getMidLandFcst` — +4~+10일 날씨·강수확률
- 발표 시각: 매일 06시, 18시 (최근 것 자동 선택)
- 5개 관측소 regId 매핑 확정:

| ASOS | 지역 | 중기기온 regId | 중기육상 regId |
| ---- | ---- | -------------- | -------------- |
| 108  | 서울 | 11B10101       | 11B00000       |
| 119  | 수원 | 11B20501       | 11B00000       |
| 133  | 대전 | 11C20101       | 11C20000       |
| 156  | 광주 | 21F20801       | 11F20000       |
| 159  | 부산 | 21F10501       | 11H20000       |

### 두 소스 예보값 차이 (실측, 2026-04-20)

- +4~+7일: 대체로 ±2℃ 범위
- **+8~+10일: 최대 ±6℃** — 후반부로 갈수록 벌어짐
- 경향: KMA가 최저 높게, 최고 낮게 (보수적)

### 이미 만든 것

- `services/api/data_pipeline/kma_mid_forecast_loader.py` — 로더 완성, 35행(5지역×7일) 반환
- 어디서도 import 안 함 → **대시보드·모델에 영향 0**

## 설계안 3가지

### 옵션 A. KMA 우선 덮어쓰기 (가장 단순)

미래 주차 피처 생성 시 +4~+10일 구간은 KMA 값으로 Open-Meteo 덮어쓰기.

**장점**: 구현 간단 (weather_loader 한 곳 수정)
**단점**: KMA가 실제로 더 정확하다는 근거 없이 도입 → **사용자 피드백 원칙 위반**
**리스크**: 겨울 MAE 악화 가능성 (평년값 실험처럼)

### 옵션 B. 앙상블 (평균)

미래 +4~+10일 구간 피처 = (Open-Meteo + KMA) / 2.

**장점**: 한쪽이 극단적으로 빗나가도 완충
**단점**: 두 모델의 bias가 누적될 수 있음. 해석 어려움.
**리스크**: 정확도 개선 불투명

### 옵션 C. 실측 비교 후 가중치 결정 (권장) ✅

1주~2주간 KMA 예보를 매일 수집 저장 → 해당 날짜 지나 실측 쌓이면 두 소스 MAE 직접 계산 → **정확한 쪽에 가중치** 부여.

**장점**: 데이터 기반 결정 (사용자 피드백 스탠스와 일치)
**단점**: 2주 대기
**리스크**: 최소 — 검증 후 도입

## 구현 범위 (옵션 C 기준)

### Phase B-1: 로그 수집 (즉시 가능)

```
services/api/data_pipeline/kma_mid_forecast_loader.py  (이미 있음)
    ↓
신규 스크립트 scripts/collect_kma_mid_daily.py
    ↓
data/processed/kma_mid_history.csv (매일 append)
    컬럼: collected_at, tm_fc, target_date, station_id, temp_min, temp_max, ...
```

- 매일 수동 또는 GitHub Actions 주기 실행
- 목적: "ECMWF vs KMA 어느 쪽이 실측과 가까운가"를 2주 뒤 비교할 데이터 축적

### Phase B-2: MAE 비교 분석 (2주 뒤)

```
data/processed/kma_mid_history.csv + asos_weather_cache.csv
    ↓
scripts/compare_forecast_vs_actual.py
    ↓
결과: {target_date_lag: MAE(ECMWF), MAE(KMA)}
       예: +4일 MAE(ECMWF)=1.8℃ vs MAE(KMA)=1.2℃
```

### Phase B-3: 통합 결정 (비교 결과 기반)

- **KMA가 유의미하게 정확하면 (예: MAE 20%+ 감소)**:
  - `weather_loader.fetch_open_meteo_ecmwf_daily` 반환 직후 +4~+10일 구간만 KMA로 교체
  - 또는 신규 `fetch_hybrid_future_weather(client)` 함수 추가
- **차이 미미하면**:
  - 로그만 유지, 통합 스킵

## 대시보드·모델 영향

Phase B-3 통합 전까지는 **0 (현 상태 유지)**. 통합 후:

- Model A 미래 예측 입력 피처가 달라짐 → 재학습·재배치 필요
- 신규 `model_version="round7-kma-mid"` 로 저장
- 대시보드는 최신 model_version 읽음 → 예측값 변화

## 롤백 전략

- 통합은 신규 model_version으로만 저장. 기존 `round5-full` 유지.
- 실측 검증에서 MAE 악화 확인되면 FastAPI 엔드포인트가 이전 model_version 읽도록 설정 변경만 하면 즉시 복귀.
